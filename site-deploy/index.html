SwiftUI + SwiftData
快速建立資料模型

這是一個標準的 CRUD 應用。透過 AI，我只需要定義出我的 Task 模型包含什麼欄位，像是名稱、開始日期、狀態與卡片顏色，它就能幫我寫好大部分的資料邏輯與畫面結構。

01
資料持久化
使用 iOS 內建的 SwiftData，不需要額外設定伺服器，資料直接存在 iPhone 本機。
02

03
UI 與配色
我直接請他使用 iOS guideline 製作，讓整體畫面在配色、留白和操作感上更貼近 iPhone 原生介面。
在主畫面直接看到目前累積天數

App 02 — 情緒紀錄
Emotion Log
需求來源

它的起點，
是一場回診

我有一段時間，每天要記錄自己吃了哪些藥、睡得怎樣、有沒有運動、情緒到哪個刻度。

因為回診的時候，醫師會問我「這個月狀況怎麼樣」，而我永遠答不出來。

可自訂欄位 月曆視覺化 健康管理

「我先用紙本記。記了一陣子之後我發現——我要的不是一個資料庫，是一個下班以後還願意打開的東西。」

紙本單日紀錄
紙本月曆紀錄

我先在紙本上把欄位和標記方式整理清楚，再把這個結構搬進 App，讓需求來源本身就能支撐後面的功能設計。

App 02 · 功能設計

不只是記錄
而是為了與醫師溝通

功能 01

睡眠狀況、運動習慣、保健食品更換記錄，作為管理身心健康所需的完整資料。

問題與解決

情緒紀錄 App · 問題與解決

兩次更新，修復兩個核心問題。因為對需求非常熟悉，整體開發順暢，主要處理了兩個具體的 Bug。

🐛
Bug 1：深色模式下藥丸消失
白色藥丸圖示在日夜間切換時，於深色背景下完全不可見，影響使用體驗。
解法：調整圖示顏色邏輯，使其依系統外觀自動切換，確保深色與淺色模式下皆清晰可見。
🐛
Bug 2：失眠標記邏輯錯誤
睡眠記錄中的「失眠標記」欄位出現邏輯判斷錯誤，無法正確對應到月曆標記顯示。
解法：重新梳理欄位選取與月曆標記的對應邏輯，修正判斷條件，讓自訂欄位正確觸發標記。

回顧與心得

做完兩個 App
我學到的事

①
需求越清楚，AI 越好用
不需要懂技術，但需要把自己的使用場景說清楚。需求模糊是所有問題的根源。
②
Codex 比 ChatGPT 更適合做開發
ChatGPT 適合諮詢，Codex 適合動手做。工具選對了，開發才會順。
③
你就是最好的 QA
自己開發、自己用，遇到問題最快。真實使用場景是最好的測試環境。
④
有時候問題不在 code
Widget 的破圖靠重裝解決。遇到怎麼改都沒用的 bug，試試換個方向。
後記

簡報做完之後，
我的資料全沒了

App 是用 Codex 開發完成的。後來我想加功能，但那時我停止訂閱 GPT 了，所以換成 Claude 接手改。純粹是預算問題，沒有想太多。

功能改好了，build 到手機上，本機資料全部消失。我沒有備份。

更糟的是後面那一步。為了修復，我照著 Claude 的建議重置了 iPhone——連帶失去 2026 上半年的所有照片。

兩次的錯誤其實是同一個：
我把「它應該知道」當成了「它一定知道」。

AI 可以替我寫完所有功能，但它不知道哪一筆資料對我來說是不能弄丟的，也不會在動手前替我算「萬一它錯了，我要賠多少」。

那是使用者才知道的事。
而我剛好是使用者。
